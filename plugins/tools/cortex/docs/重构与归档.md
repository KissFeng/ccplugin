# 重构与归档

本文回答：cortex-refactor 4 个子操作 (rename / merge / split / fold) 怎么用、backup 路径在哪、有什么不可逆风险。
适用读者：要改 vault 结构的进阶用户、想批量整理旧笔记的 PKM 老手。

## 总览

| 子操作   | 触发短语                                            | 是否需 `--apply` | backup |
| -------- | --------------------------------------------------- | ---------------- | ------ |
| `rename` | 显式: `"cortex refactor rename '<old>' '<new>'"`    | 是               | 是     |
| `merge`  | 显式: `"cortex refactor merge '<src>' '<dst>'"`     | 是               | 是     |
| `split`  | 显式: `"cortex refactor split '<file>' --by h2"`    | 是               | 是     |
| `fold`   | 显式 (`refactor fold`) 或自动 (`` skill) | 是               | 是     |

cortex-refactor 是显式触发 (`disable-model-invocation: true`), 大动干戈, 必须用户明确请求。

默认 dry-run, 输出预览 diff。`--apply` 才实际改盘并 backup。

## backup 机制

任何 `--apply` 操作前, 先把受影响文件原版拷到：

```
<vault>/.cortex-backup/<YYYYMMDD-HHMMSS>/
├── <relative-path>/<file>.md
└── manifest.json   # 列出所有备份文件 + 操作类型
```

回滚：手动从 `.cortex-backup/<timestamp>/` 拷回。cortex 不提供 `undo` 命令, 因为反链可能已被其他写入更新。

## rename

### 用途

改文件名, 同时修所有反链 (wikilink + frontmatter `related` / `up` / `sources`)。

### 示例

```text
"cortex refactor rename '知识库/领域/事件驱动.md' '知识库/领域/事件驱动架构.md'"
```

dry-run 输出：

```text
将改名: 知识库/领域/事件驱动.md → 知识库/领域/事件驱动架构.md
将更新反链 (3 处):
  - 知识库/项目/github.com/lazygophers/ccplugin/desktop.md (line 42)
  - _assets/dashboards/concepts-dashboard.md (line 18)
  - hot.md (line 7)
```

确认后加 `--apply`：

```text
"cortex refactor rename '...' '...' --apply"
```

### 风险

- 若有外部链接 (Obsidian publish 站、其他工具) 引用旧路径, 不会被修。
- alias 列表中含旧名时, 不会自动加到新文件 (避免歧义)。

## merge

### 用途

合并两页, 旧页变重定向 (frontmatter `aliases` 加旧名 + body 留链)。

### 示例

```text
"cortex refactor merge '知识库/领域/CQRS.md' '知识库/领域/事件溯源.md'"
```

dry-run 输出：

```text
源: 知识库/领域/CQRS.md (37 行, 5 个反链)
目标: 知识库/领域/事件溯源.md (82 行)

操作:
  1. 在目标末尾追加源内容, 加分隔标题 "## (合并自 CQRS)"
  2. 源 frontmatter 改为: aliases: [CQRS], type: redirect
  3. 源 body 替换为: "已并入 [[事件溯源]]"
  4. 5 个反链不动 (alias 还在生效)
```

### 风险

- 内容长度叠加可能超 lint `log-too-long` 阈值。
- 源页若有重要 block-id, 跨文件引用会失效。建议 merge 前 grep 一下 `^cortex-` 与 `#^cortex-`。

## split

### 用途

把一页按 H2 拆成多页, 原页变索引 (列拆分后的子页 wikilink)。

### 示例

```text
"cortex refactor split '知识库/领域/微服务全栈.md' --by h2"
```

dry-run 输出：

```text
源: 知识库/领域/微服务全栈.md (有 6 个 H2)

将拆为:
  1. 知识库/领域/微服务全栈/服务发现.md
  2. 知识库/领域/微服务全栈/熔断与降级.md
  3. 知识库/领域/微服务全栈/链路追踪.md
  ...

源页改为索引页, body 列 6 个 wikilink。
```

### 风险

- 拆分后子页缺独立背景, 阅读体验下降。建议先在源页确认每个 H2 自包含。
- 反链失效：原本指向 `[[微服务全栈#服务发现]]` 的链接需手动改 `[[微服务全栈/服务发现]]`。

## 自动化 (cron)

归档由 digest 自动处理 (详见 `周期任务.md`)。

## 不可逆风险清单

| 操作   | 风险等级 | 说明                                  |
| ------ | -------- | ------------------------------------- |
| rename | 中       | backup 可回滚, 但反链已散落难手动还原 |
| merge  | 高       | 内容合并后人工拆开成本高              |
| split  | 高       | 反链需手动改, 拆分语义不易回退        |

**铁律**：

1. 跑 `--apply` 前先 dry-run 看预览。
2. 跑 `--apply` 前先 `git commit` (或 OGit 自动 commit)。
3. backup 至少保留 30 天再清理 (`cortex-doctor` 会提醒)。
